# Подготовка к Simulink

## Расчетное ядро

Расчетное ядро отделено от чтения файлов, построения графиков и формирования отчетов. Это позволяет использовать его как основу для будущих функциональных блоков MATLAB в Simulink.

## Функции, пригодные для переноса

- `rhsRateModel`;
- `rhs6DOF`;
- `MotorModel`;
- `BatteryModel`;
- `coordinateSystems`.

## Функции, которые нельзя переносить в Simulink

К препроцессору и постпроцессору относятся:

- чтение файлов;
- формирование отчетов;
- построение графиков;
- запись JSON;
- расчет хэшей.

Эти операции должны оставаться вне расчетных блоков.

## Интерфейс модели угловых скоростей

Входы:

- `omega = [p; q; r]`, рад/с;
- `u = [u_roll; u_pitch; u_yaw]`;
- параметры `A`, `B`, `c`.

Выход:

- `domega/dt`, рад/с².

## Интерфейс модели движения с шестью степенями свободы

Входы:

- состояние: положение, скорость, ориентация, угловые скорости, тяга двигателей;
- нормированные команды двигателей;
- параметры массы, инерции, геометрии, винтомоторной группы и сопротивления воздуха.

Выход:

- производная состояния.

Интерфейсы входов и выходов должны сохраняться при дальнейшем переносе в Simulink.

## Подготовка PR №5 к будущему Simulink

В PR №5 добавлены расчетные функции, которые могут стать основой отдельных блоков Simulink:

- `rhs6DOF` как блок объекта моделирования `Model6DOF plant`;
- `PrimitiveAttitudeController` как примитивный блок углового управления;
- `PrimitiveAltitudeController` как примитивный блок высотного управления;
- `PrimitivePositionController` как внешний контур положения;
- `quadXMixer` как смеситель QUAD/X;
- `MotorModel` как заготовка модели винтомоторной группы.

Эти функции принимают параметры, состояния и входы явно. Они не должны читать файлы, строить графики или формировать отчеты.

Постпроцессор визуализации переносить в Simulink не требуется. `plotLogOverview`, `plotFlightPath3D`, `drawCopterFrame`, `animateCopterFlight` и сценарии PR №5 относятся к анализу результатов и демонстрации.

Полноценная модель Simulink будет отдельным следующим этапом. Текущий PR проверяет, что расчетное ядро, примитивный регулятор и постпроцессор имеют понятные интерфейсы для будущей декомпозиции на блоки.

## Связь с replay Model6DOF PR №6

Перенос в Simulink должен выполняться после подтверждения устойчивого replay-контура в MATLAB. Уже выделены функции, которые сохраняют четкие входы и выходы и могут стать основой будущих блоков:

- `rhs6DOF`;
- `MotorModel`;
- `quadXMixer`;
- `PrimitiveAttitudeController`;
- `PrimitiveAltitudeController`;
- `PrimitivePositionController`;
- `simulatePrimitiveClosedLoop`.

Функции чтения файлов, формирования отчетов, построения графиков и записи JSON к переносу в Simulink не относятся.

## Условие переноса после PR №7

Перенос `Model6DOF` в Simulink допускается только после фиксации:

- motor map;
- знаков вращения моторов;
- знака тяги;
- знака момента рыскания;
- преобразования `CAD -> body`;
- нормирования `RCOU`;
- набора параметров ArduPilot, влияющих на входы двигателей.

В будущий блок `Plant6DOF` должны передаваться:

- масса;
- диагональная матрица инерции или подтвержденный полный тензор;
- координаты моторов в связанной системе координат;
- знаки вращения;
- `kT`, `tau_motor_s`, `kQ_over_kT`;
- коэффициенты линейного сопротивления;
- нормированные команды двигателей в подтвержденном физическом порядке.

PR №7 не выполняет перенос в Simulink. Он устраняет риск переноса неверной физической конфигурации в отдельную среду расчета.

## Статусы перед переносом Plant6DOF

Перенос `Plant6DOF` в Simulink допускается только после фиксации статусов:

- `motor_map_status`;
- `spin_sign_status`;
- `cad_to_body_status`;
- `propulsion_model_status`;
- `propeller_configuration_status`.

Если любой из этих статусов остается `hypothesis_from_log`, `hypothesis_only`, `demonstration` или `inconsistent_sources`, Simulink-модель допускается только как демонстрационная. PR №8 не выполняет перенос в Simulink и готовит данные управления конфигурацией для будущего этапа.
