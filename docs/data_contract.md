# Договор на данные

## Форматы первого этапа

Поддерживаются `CSV`, `MAT`, `table` MATLAB и `timetable` MATLAB. В PR №2 добавлен анализатор бортовых журналов ArduPilot DataFlash `.BIN` для выявления состава сообщений и подготовки адаптера чтения. Окончательная идентификация параметров по `.BIN` выполняется следующим этапом после подтверждения состава каналов.

## Обязательные поля для модели угловых скоростей

- `TimeUS` или другой столбец времени;
- `RATE.R`;
- `RATE.P`;
- `RATE.Y`;
- `RATE.ROut`, `RATE.POut`, `RATE.YOut` или `RCOU.C1-C4`.

## Необязательные поля

- `ATT.Roll`, `ATT.Pitch`, `ATT.Yaw`;
- `ATT.DesRoll`, `ATT.DesPitch`, `ATT.DesYaw`;
- `RATE.RDes`, `RATE.PDes`, `RATE.YDes`;
- `RATE.AOut`;
- `BAT.Volt`, `BAT.Curr`, `BAT.CurrTot`, `BAT.EnrgTot`;
- `ESC.RPM`, `ESC.Curr`, `ESC.Volt`.

Отсутствие необязательных полей не должно аварийно завершать выполнение препроцессора, решателя или постпроцессора.

## Требования к времени

Допустимые столбцы времени: `TimeUS`, `TimeMS`, `TimeS`, `Time`, `timestamp`, `t`, `time`.

Препроцессор приводит время к секундам, сортирует строки, удаляет дубли и приводит данные к равномерной сетке.

## Требования к единицам измерения

- Углы `ATT.*` ожидаются в градусах и приводятся к радианам.
- Угловые скорости `RATE.*` ожидаются в градусах в секунду и приводятся к радианам в секунду.
- `RCOU.C1-C4` ожидаются в микросекундах и нормируются по `pwm_min` и `pwm_max`.

## Требования к именам столбцов

Имена столбцов должны сохранять имена полей бортового журнала ArduPilot, например `RATE.R` и `RCOU.C1`.

## Бортовые журналы ArduPilot DataFlash .BIN

Файлы размещаются локально в каталоге:

```text
data/raw/ardupilot/bin/
```

Рабочие имена журналов первого анализа:

- `data/raw/ardupilot/bin/VB-01.alt_50m.BIN`;
- `data/raw/ardupilot/bin/full_fly_1.BIN`;
- `data/raw/ardupilot/bin/full_fly_2.BIN`.

Ключевые сообщения для первого этапа анализа состава каналов:

- `FMT` для описания форматов сообщений;
- `PARM`, `MSG`, `MODE` для контекста полета и настройки изделия;
- `ATT`, `RATE`, `RCOU` для модели угловых скоростей;
- `BAT`, `ESC`, `ESC1`, `ESC2`, `ESC3`, `ESC4` для модели аккумулятора и винтомоторной группы;
- `GPS`, `POS`, `XKF`, `XKF1`, `XKF2`, `XKF3`, `XKF4` для дальнейшей подготовки данных `Model6DOF`;
- `IMU`, `ACC`, `GYR`, `BARO` для анализа качества исходных данных и границ применимости.

Для модели угловых скоростей обязательны сообщения `RATE` и источники управляющих воздействий `RATE.ROut`, `RATE.POut`, `RATE.YOut` либо `RCOU`. Сообщение `ATT` желательно для сопоставления углов и дальнейшей оценки адекватности.

Для `Model6DOF` желательны `ATT`, `RATE`, `RCOU`, `GPS`, `POS`, `XKF*`, `IMU`, `ACC`, `GYR`, `BARO`. На PR №2 эти данные только выявляются и приводятся к реестру каналов; окончательная идентификация параметров не выполняется.

Для модели аккумулятора нужны `BAT` и, при наличии, `ESC` или `ESC1`...`ESC4`. Отсутствие этих сообщений фиксируется предупреждением, но не должно аварийно останавливать анализ журнала.

Для дальнейшей оценки адекватности по ВБ нужны сообщения, позволяющие выделить режим, участок времени, входные воздействия и проверяемые параметры: `MODE`, `ATT`, `RATE`, `RCOU`, `BAT`, `GPS`, `POS`, `XKF*`. Текущий уровень остается анализом бортовых журналов и подготовкой данных для оценки адекватности компьютерной модели.

Сырые `.BIN` журналы могут содержать чувствительную информацию. Их не следует добавлять в обычный Git; для хранения в репозитории нужно использовать Git LFS или закрытый репозиторий.
