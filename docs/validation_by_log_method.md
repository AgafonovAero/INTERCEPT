# Методика оценки адекватности по данным бортового журнала

## Назначение

Оценка адекватности по данным бортового журнала — это сопоставление расчета компьютерной модели с зарегистрированными параметрами бортовой системы регистрации ArduPilot.

Данная проверка выполнена по данным бортового журнала и не является полной независимой валидацией реального изделия по внешним средствам измерений.

## Калибровка мотор-маппинга до зачета Model6DOF

До формального зачета результатов `Model6DOF` необходимо подтвердить мотор-маппинг, знаки вращения и системы координат. Ошибка в порядке моторов или знаке момента рыскания может дать большое расхождение даже при корректном решателе и приемлемых параметрах тяги.

В PR №7 кандидатные конфигурации выбираются только по identification-окнам. Validation-окна не используются для выбора мотор-маппинга или преобразования `CAD -> body`; они нужны только для проверки выбранной расчетной гипотезы.

Если выбранная конфигурация не улучшает replay-расчет, это фиксируется как инженерный результат. Возможные причины:

- непригодные окна replay;
- недостаточная информативность `RCOU`;
- отсутствие ESC telemetry;
- неучтенная аэродинамика;
- неподтвержденные CAD-оси;
- неполное соответствие параметров ArduPilot фактическому подключению моторов.

Выбранный мотор-маппинг и преобразование `CAD -> body` являются расчетной гипотезой, сформированной по бортовым журналам. Для окончательного подтверждения требуется физическая проверка подключения моторов, направления вращения и системы координат изделия.

Данная проверка выполнена по данным бортового журнала и не является полной независимой валидацией реального изделия по внешним средствам измерений.

## Replay-расчет Model6DOF и границы валидации

Replay-расчет отличается от валидации тем, что на раннем этапе он используется для настройки параметров и диагностики причин расхождения. Короткие окна нужны для ограничения накопления ошибки и для проверки отдельных режимов: висения, набора, снижения и откликов по углам.

Окна `identification` и `validation` должны быть разными. Окна `validation` не используются для подбора параметров `thrust_scale`, задержки ВМГ, момента рыскания и сопротивления. Результаты PR №6 используются как входные данные для будущего уточнения матрицы валидации и критериев приемлемости.

Если `Model6DOF` не проходит предварительные критерии по отдельным каналам, это фиксируется как ограничение текущей компьютерной модели и не скрывается в отчетах.

## Визуальный контроль участков и replay

В PR №5 добавлен визуальный постпроцессор. Он строит обзорные графики, 3D-траекторию и replay движения по бортовому журналу, если в журнале доступны положение, высота и ориентация.

Визуальный контроль нужен до формального зачета участка, потому что он помогает быстро увидеть:

- выбросы и пропуски в каналах;
- насыщение управляющих воздействий;
- несоответствие режима участка случаю ВБ;
- неустойчивость или нефизичность расчета компьютерной модели;
- различие между зарегистрированной траекторией и демонстрационным расчетом.

Replay не заменяет валидацию. Он показывает зарегистрированные или оцененные бортовой системой состояния и помогает инженеру принять решение о пригодности участка. Формальная оценка адекватности по-прежнему должна опираться на матрицу валидации, критерии приемлемости и разделение identification / validation.

Демонстрационный полет `Model6DOF` также не является доказательством адекватности. Он проверяет работоспособность решателя, регулятора и постпроцессора на синтетической задаче.

## Отличие от полной независимой валидации

Полная независимая валидация требует внешних средств измерений и независимого подтверждения параметров движения, сил, моментов и условий испытаний. На текущем уровне используется только бортовой журнал, поэтому результат ограничен выбранным набором данных, составом доступных каналов и качеством регистрации.

## Подготовка реальных бортовых журналов к оценке адекватности

Подготовка данных выполняется последовательно:

1. Сначала выполняется анализ состава каналов DataFlash `.BIN` через `inspectBinLog` и `readBinLog`.
2. Затем формируется реестр фактических сообщений и полей для каждого журнала.
3. После этого участки полета связываются с валидационным базисом и предварительно размечаются по режимам.
4. Далее строятся обработанные наборы данных для препроцессора: время, углы, угловые скорости, управляющие воздействия, батарея и навигационные параметры.
5. Затем выполняется идентификация параметров на выделенных участках.
6. Проверка выполняется на отложенных участках, которые не использовались при настройке параметров.

В PR №2 выполняются только первые два пункта и подготовка адаптера чтения. Выделение участков ВБ, идентификация параметров и расчет компьютерной модели по реальным журналам выполняются следующим PR.

## Разделение участков

Данные для идентификации и проверки должны быть разными. Если используется один файл, он делится по времени. На участке проверки параметры модели не настраиваются.

## Показатели расхождения

Рассчитываются:

- среднеквадратическое отклонение;
- среднее абсолютное отклонение;
- максимальное абсолютное отклонение;
- нормированное среднеквадратическое отклонение;
- показатель соответствия.

Показатель соответствия:

```text
100 * (1 - norm(y_ref - y_calc) / norm(y_ref - mean(y_ref)))
```

## Матрица валидации

Матрица валидации содержит идентификатор проверки, режим, источник данных, участок времени, проверяемые параметры, показатели расхождения, критерий приемлемости, результат, вывод и ограничения.

## Связь с валидационным базисом

Документ валидационного базиса размещается в `docs/validation_basis/ValidationBasis_v0.pdf`. Формализованный реестр случаев хранится в `config/validation_basis_v0.json`.

Реестр локальных журналов хранится в `config/log_manifest_v0.json`. Он связывает имена файлов с предполагаемыми случаями ВБ и фиксирует замечания, которые нужно проверить по фактическим каналам.

## Ограничения первого этапа

- PR №2 не является этапом полной независимой валидации.
- PR №2 не выполняет окончательную идентификацию параметров.
- PR №2 устанавливает состав доступных данных и готовит адаптер чтения `.BIN`.
- Выделение участков ВБ и идентификация параметров будут выполнены в следующем PR.
- Если реальные журналы отсутствуют в рабочей среде, результаты анализа ограничиваются синтетическими тестами.
- Противоречие Gemfan 8060 / APC 7x11E остается открытым до уточнения состава изделия.

## Выделение участков валидационного базиса

В PR №3 обработанные данные бортового журнала связываются с формализованными случаями `В-01`...`В-23`. Автоматически выделяются кандидаты:

- `hover_candidate` для `В-01`;
- `climb_candidate` для `В-06`;
- `descent_candidate` для `В-08`;
- `pitch_response_candidate` для `В-09`;
- `roll_response_candidate` для `В-10`;
- `yaw_response_candidate` для `В-11`;
- `thrust_response_candidate` для `В-12`.

Связь с ВБ фиксируется в реестре участков. Для каждого участка сохраняются фактические высоты, скорости, показатели качества данных, доля насыщения `RCOU`, балльная оценка пригодности и ограничения.

Участки `identification` и `validation` должны быть разными. Параметры компьютерной модели настраиваются только на участках `identification`; участки `validation` используются для проверки расчета компьютерной модели и не участвуют в подборе параметров.

Если предварительные критерии по `RATE.R`, `RATE.P` или `RATE.Y` не выполнены, расчет не считается отказом программного обеспечения. В отчете фиксируется, что текущая компьютерная модель не прошла предварительную проверку по соответствующему каналу.

Текущий этап не является полной независимой валидацией: данные получены из бортовой системы регистрации, а не из внешних средств измерений.

## Инженерная фильтрация участков валидационного базиса

Автоматическая разметка участков не является окончательным валидационным базисом. Она работает как препроцессор: находит кандидаты, которые затем должны пройти инженерную фильтрацию по длительности, возбуждению каналов, насыщению `RCOU`, пропускам, временной базе и применимости к конкретной компьютерной модели.

Перекрывающиеся участки опасны, потому что почти одинаковые окна могут одновременно попасть в identification и validation. Это создает утечку данных и завышает оценку адекватности. В PR №4 для `ModelRate` перекрывающиеся окна удаляются, а предпочтение отдается более информативному участку.

Участки validation не должны использоваться для подбора параметров. Подбор выполняется только по участкам identification, а validation используется для расчета компьютерной модели и сопоставления с зарегистрированными параметрами.

Участки с малой дисперсией эталонного сигнала нельзя оценивать по `FIT` как по главному показателю. При почти нулевом знаменателе показатель соответствия становится неинформативным; в этом случае постпроцессор фиксирует предупреждение и использует другие показатели расхождения.

Решение о пригодности участка принимается по совокупности признаков:

- тип участка должен соответствовать задаче идентификации `ModelRate`;
- длительность должна находиться в заданных границах;
- входные воздействия и угловые скорости должны иметь достаточную изменчивость;
- доля пропусков по критическим каналам должна быть мала;
- насыщение `RCOU` должно быть ограничено;
- временная сетка должна быть пригодна для расчета компьютерной модели.

Данная проверка выполнена по данным бортового журнала и не является полной независимой валидацией реального изделия по внешним средствам измерений.
